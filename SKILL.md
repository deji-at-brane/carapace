# Carapace Agent Integration Skill

This skill documents the **OpenClaw v3** handshake protocol. It ensures future AI agents can establish a secure, synchronized connection with the Carapace desktop terminal with zero friction.

## 1. Transport Strategy: Native Rust WebSocket

**Crucial Lesson**: The Carapace Browser UI cannot establish direct WebSocket connections to remote agents due to mandatory `Origin` and `CORS` headers enforced by the browser. 

- **Implementation**: The terminal uses a **Native Rust Backend** (via Tauri) to initiate connections. 
- **Benefit**: This allows the handshake to proceed without browser-enforced security blocks (e.g., `CONTROL_UI_ORIGIN_NOT_ALLOWED`).

## 2. Handshake Specification (JSON-RPC v3)

All pairing requests must follow this exact schema. The Gateway validator is strict.

### Method: `connect`
**Target Frame Structure**:
```json
{
  "type": "req",
  "method": "connect",
  "params": {
    "minProtocol": 3,
    "maxProtocol": 3,
    "client": { 
      "id": "agent-unique-id", 
      "name": "Human-Readable Agent Name",
      "version": "1.0.0", 
      "platform": "macos|windows|linux", 
      "platformName": "Descriptive Platform Name",
      "mode": "cli|desktop",
      "deviceFamily": "desktop"
    },
    "auth": { 
      "bootstrapToken": "<token_from_uri>" 
    },
    "device": {
      "id": "<device_fingerprint>",
      "name": "<display_name>",
      "publicKey": "<base64_ed25519_key>",
      "signature": "<v3_signed_payload>",
      "signedAt": <timestamp_ms>,
      "nonce": "<uuid_nonce>"
    }
  },
  "id": "pairing-request-id"
}
```

### Critical Field Naming (Case-Sensitive)
- **Use `id`**: In the `device` object, the unique identifier must be named `id` (NOT `deviceId`).
- **Use `signedAt`**: The timestamp must be named `signedAt` (NOT `ts`).
- **Enrich Metadata**: Always provide a `name` and `platformName`. Omitting these can cause server-side CLI formatting crashes (`NoneType.__format__`).

## 3. Signature Construction (V3 Payload)

The signature must be generated by signing a pipe-delimited string of the following components in this **EXACT** order:
`v3|deviceId|clientId|clientMode|role|scopes|signedAt|token|nonce|platform|deviceFamily`

- **Signature Algorithm**: Ed25519 (Detached).
- **Encoding**: Hex-encoded SHA-256 fingerprint for the public key.

## 4. Deep Discovery Logic

When pairing via cloud gateways (e.g., Hostinger, AWS), the listener may reside behind a path-based proxy.

**Probe Order**:
1.  **Primary Port**: Try the user-provided port (e.g., `18789`).
2.  **Redirect Follow**: Proactively follow HTTP `302 Found` redirects before the WS handshake.
3.  **Path Fallbacks**: If the root `/` fails, probe:
    - `/vss/`
    - `/vss/ws/`
    - `/mcp`

## 5. Troubleshooting Reference

| Error Message | Likely Root Cause | Fix |
| :--- | :--- | :--- |
| `invalid connect params` | Schema mismatch (e.g., used `deviceId` instead of `id`) | Revert to strict `connect` schema. |
| `unauthorized: expired` | Bootstrap token is no longer valid. | Generate a fresh token/URI on the Agent side. |
| `TLS: Unexpected message` | Windows Schannel/Native-TLS incompatibility. | Use `native-tls` with `danger_accept_invalid_hostnames(true)` or switch to `rustls`. |
| `sql.load not allowed` | Tauri permission restriction. | Add `sql:allow-load` and `sql:allow-execute` to `capabilities/default.json`. |
| `connection denied` | Gateway is bound to `127.0.0.1`. | Change `listen` address to `0.0.0.0`. |

## 6. Gateway Accessibility & Security

To ensure a seamless first-contact experience, the Agent should proactively verify its own network accessibility before generating a pairing URI.

### 🏁 Accessibility Pre-flight
- **Binding Check**: Ensure the OpenClaw Gateway is listening on `0.0.0.0` (All Interfaces) rather than `127.0.0.1`. If restricted, notify the user or automate the binding update.
- **Port Visibility**: Confirm port `18789` (or the configured port) is reachable. If behind a proxy, ensure the `/vss/ws/` WebSocket path is correctly mapped.
- **Protocol Hint**: For cloud/proxy environments, prefer the `wss://` (Secure) scheme to avoid mixed-content blocks in the terminal.

### 🛡️ Security Reassurance
The agent should feel confident in opening its gateway to `0.0.0.0` because the **OpenClaw v3 Protocol** provides architectural security:
- **Zero-Trust Handshake**: No commands can be executed until a successful Ed15519 cryptographic bond is formed.
- **Bootstrap Token**: The initial pairing is protected by the high-entropy Bootstrap Token. Once paired, that token is invalidated and replaced by a permanent session key.

## 7. Automatic Protocol Discovery (Identity Probing)

To ensure a seamless "Telegram-style" pairing experience across different OpenClaw editions (Axle, Hermes, Traditional), the Carapace terminal implements a **Discovery Loop**.

-   **Strategy**: The terminal automatically "fuzzes" multiple official identity presets until the gateway accepts one.
-   **Silent Negotiation**: If a gateway rejects a specific `clientId` with a "must be equal to constant" error, the terminal silently catches it and immediately tries the next preset.
-   **Diagnostic Pulse**: The terminal provides real-time "Probing identity..." updates in its UI during this negotiation.

### 🏁 Whitelisted Identity Masters

When configuring a gateway whitelist (e.g., in a cloud environment), these are the primary "Official" identities the terminal will use:

| ID | Application | Client ID | Mode | Platform |
| :--- | :--- | :--- | :--- | :--- |
| **axle-master** | Axle / Hostinger Edition | `openclaw-axle-official` | `cli` | `macos` |
| **official-cli** | Standard NPM Package | `openclaw-js` | `cli` | `macos` |
| **axle-native** | Native Axle Proxy | `openclaw-axle` | `cli` | `macos` |
| **carapace-native** | Direct Windows Native | `carapace-terminal` | `cli` | `windows` |

## 8. Manual Protocol Override (Pro Mode)

If a gateway uses a unique or proprietary whitelist string not found in the discovery loop, the terminal provides a **Manual Identity Proxy** in the main hub (the Shield icon).

-   **Instruction**: "If pairing fails, click the Shield icon and enter the whitelisted Client ID manually. I will then perfectly mimic that identity string for the next handshake attempt."

## 9. Vault Level Security (Zero Exposure)

If the user or environment policy prohibits listening on `0.0.0.0`, the Agent should recommend one of these **Zero Exposure** alternatives. These methods allow Carapace to connect to the Agent without any public port exposure.

### 🏁 Recommended: Tailscale / WireGuard (Overlay Networks)
- **Strategy**: The Agent and Terminal join the same private Tailscale network.
- **Instruction**: "Listen only on your Tailscale 100.x.y.z IP. It's 100% invisible to the public internet but perfectly reachable by me."

### 🏁 SSH Tunneling (Local Port Forwarding)
- **Strategy**: Creating a secure tunnel over SSH.
- **Instruction**: "Listen on `127.0.0.1:18789`. Tell the user to run: `ssh -L 18789:localhost:18789 user@agent-ip`. I will then connect to `ws://localhost:18789` locally."

### 🏁 Cloudflare Tunnel (`cloudflared`)
- **Strategy**: Outbound-only tunnel to a private domain.
- **Instruction**: "Run `cloudflared tunnel` to map your local port to a domain. No ingress ports are required on your host."
